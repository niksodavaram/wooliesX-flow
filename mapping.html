C · MIGRATION PLAN
How I’d approach Windows VM → RHEL MicroShift for 3,000+ stores
18-month roadmap

This is not a perfect project plan; it’s a high-level sequence that shows how I think about risk, pilot stores, canary waves and compliance when replacing fragile Windows VMs with an immutable RHEL + MicroShift stack.

  • Phase 1 — Design & 5-store pilot
    Weeks 1–4
    Design the RHEL 9 Image Builder blueprint and LVM layout, stand up the ACM hub and Git repo structure, and define Kustomize overlays per store tier. Run RHEL + MicroShift in parallel with the existing Windows VM in 5 pilot stores.
  • Phase 2 — Validate flows & harden
    Weeks 5–8
    Prove end-to-end: POS → Kafka → BigQuery, inventory → SAP S/4HANA, and cold-chain MQTT → Vertex AI. Only when all flows are green and CIS/PCI-DSS baselines pass do we consider Windows removable.
  • Phase 3 — Canary waves at 50 stores/week
    Months 3–6
    Roll out in waves with ArgoCD, each wave ~50 stores. Health gates and SLOs drive automated rollback if error rates spike. ACM dashboards give fleet-wide compliance and drift visibility before retiring the Windows VM per store.
  • Phase 4 — Full fleet + ZTP by default
    Months 7–18
    All stores run on RHEL + MicroShift, Windows VMs fully decommissioned. New stores boot via zero-touch provisioning from the golden image, and we layer in MCP-style AI self-healing across the fleet.
