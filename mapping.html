A · MY 4 STORIES
Why each of these four roles matters for this job
Scale, resilience, compliance

I’ve grouped 15+ years into four anchor stories that match WooliesX’s problem space: running a large, regulated, distributed edge fleet that must keep trading when WAN, hardware or upstream systems fail.

NAB — 800 branches
Edge Fleet Engineer · containerised devices, Rancher, F5/NGINX/HAProxy
Managed ATMs, TCRs, card-swipe, passbook printers and cash machines across ~800 branches. All ran containerised microservices on hardened Ubuntu, orchestrated centrally via Rancher with traditional load balancers in front.
Why this matters for WooliesX: proved I can operate a regulated edge fleet at scale. 800 NAB branches → 3,000 Woolies stores is the same hub-and-spoke pattern, just larger.
Energy Australia CDR
Greenfield Platform Engineer · Azure, AKS, CI/CD, policy-as-code
Built the ACCC-mandated Consumer Data Right platform from zero in 8 months, on Azure/AKS. Hybrid control plane (central policy + distributed clusters), GitHub Actions/Azure Pipelines, and policy-as-code gates on every pipeline.
Why this matters for WooliesX: shows I can deliver a greenfield, compliance-heavy platform under a hard date — similar to a time-boxed Windows → RHEL MicroShift migration.
Saab Australia — 9LV warship edge
Edge Infra · NV1 cleared · real-time systems
Owned the edge infrastructure layer for the 9LV Combat Management System on Hunter-class frigates. Ubuntu/Debian nodes connect sonar, radar and other sensors to tactical displays via RTI DDS, under ISM/ASD, STIGs and CIS hardening.
Why this matters for WooliesX: a warship must keep fighting when cut off from command. That’s the same offline-first requirement as a store that must keep trading when SD-WAN drops.
CCG / Vic Uni — LEON Linux field stations
Distributed Edge Systems · PhD-backed work
Built 16 LEON Linux base stations (Xilinx ML506 + Aeroflex Gaisler SPARC) across remote NZ/Alaska, tracking 3,000 tagged birds. Edge nodes ran local Bayesian inference and synced back via rsync/SSH when links came up, with Python/Bash/cron automation.
Why this matters for WooliesX: designed offline-first Linux nodes that self-healed with zero on-site support — exactly the operating model for 3,000 stores with MicroShift and ACM.

C · MIGRATION PLAN
How I’d approach Windows VM → RHEL MicroShift for 3,000+ stores
18-month roadmap

This is not a perfect project plan; it’s a high-level sequence that shows how I think about risk, pilot stores, canary waves and compliance when replacing fragile Windows VMs with an immutable RHEL + MicroShift stack.

    >
    Phase 1 — Design & 5-store pilot
    Weeks 1–4
    Design the RHEL 9 Image Builder blueprint and LVM layout, stand up the ACM hub and Git repo structure, and define Kustomize overlays per store tier. Run RHEL + MicroShift in parallel with the existing Windows VM in 5 pilot stores.
    >
    Phase 2 — Validate flows & harden
    Weeks 5–8
    Prove end-to-end: POS → Kafka → BigQuery, inventory → SAP S/4HANA, and cold-chain MQTT → Vertex AI. Only when all flows are green and CIS/PCI-DSS baselines pass do we consider Windows removable.
    >
    Phase 3 — Canary waves at 50 stores/week
    Months 3–6
    Roll out in waves with ArgoCD, each wave ~50 stores. Health gates and SLOs drive automated rollback if error rates spike. ACM dashboards give fleet-wide compliance and drift visibility before retiring the Windows VM per store.
    >
    Phase 4 — Full fleet + ZTP by default
    Months 7–18
    All stores run on RHEL + MicroShift, Windows VMs fully decommissioned. New stores boot via zero-touch provisioning from the golden image, and we layer in MCP-style AI self-healing across the fleet.
